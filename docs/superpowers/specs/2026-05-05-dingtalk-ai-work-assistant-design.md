# 钉钉 AI 个人工作助手设计

日期：2026-05-05

## 背景

用户希望借助大模型能力打造一个项目管理/日常工作助手。第一阶段重点服务个人管理工作，不做团队级项目协作平台。工作事项主要来自钉钉聊天、邮件和临时消息，第一版采用手动复制、粘贴或转发给钉钉助手的方式，不自动读取邮箱或聊天记录。

这个助手的核心价值不是提醒待办，而是在记录任务或问题时主动对话，帮助用户澄清目标、时间节点、风险、协作人、下一步，并针对算法研发工作给出技术维度的建议或提醒。

## 目标

第一阶段目标是构建一个部署在国内云服务器上的钉钉 AI 个人工作助手：

- 用户可以在钉钉里向助手发送工作事项、问题、进展、邮件正文或临时消息。
- 助手能识别消息意图，结构化记录事项，并在信息不足时主动追问。
- 助手能体现轻量自主思考，包括追问澄清、判断建议和复盘督办。
- 助手能识别算法研发相关事项，并提示数据、指标、实验、上线、回滚、复现等技术风险点。
- 助手每天上班推送今日关注事项，每天下班推送当日记录汇总和明日关注点。
- 数据和模型服务优先使用国内云和国内模型。

## 非目标

第一阶段不做以下能力：

- 自动读取用户邮箱、钉钉聊天记录或其他 IM 消息。
- 团队级权限、项目空间、复杂看板、多人协作流程。
- 自动修改代码、查询日志平台、接入实验平台或代码仓库。
- 完整 Web 管理后台。
- 依赖海外模型作为默认链路。

这些能力可以在 MVP 验证后作为后续阶段扩展。

## 推荐方案

采用“钉钉机器人 + 后端服务 + 数据库 + 国内大模型”的自研 MVP。

对比方案：

- 钉钉宜搭 AI 任务督办适合快速验证任务督办，但更像流程化任务管理，不足以满足“自主思考助手”的核心需求。
- 低代码或 Agent 平台能加快搭建，但对任务状态、对话策略、早晚报和算法研发语境的个性化控制较弱。
- 自研轻量后端复杂度可控，也最适合沉淀自己的追问策略、任务模型和复盘逻辑。

## 系统架构

系统由以下模块组成：

- 钉钉消息接入模块：接收用户私聊或群内消息，校验钉钉回调签名，并把消息传给后端处理流程。
- 对话与任务理解模块：判断消息是新事项、进展更新、风险阻塞、查询、复盘请求、纠错还是闲聊。
- 模型调用适配层：默认接入通义千问/Qwen 或阿里云百炼，保留 Provider 抽象，后续可评估 DeepSeek、智谱、豆包等国内模型。
- 任务数据库：保存结构化事项、原始输入、状态、摘要和更新时间。
- 追问与建议策略模块：根据缺失字段、风险等级和事项类型决定是否追问、追问什么、是否给建议。
- 早晚报定时任务：每天上班生成今日关注，每天下班生成当日总结。
- 日志与错误处理模块：记录请求 ID、错误类型和必要调试信息，避免敏感内容进入日志。

## AI 行为策略

助手每次收到消息后执行四步判断：

1. 识别类型：判断消息是新任务、已有任务进展、风险或阻塞、查询、复盘请求、纠错还是闲聊。
2. 抽取结构：从原文抽取标题、背景、目标、截止时间、相关人、下一步、风险、状态。
3. 判断缺口：识别是否缺目标、时间、责任边界、依赖对象、验收标准或下一步。
4. 选择回应方式：追问、建议、更新记录、提醒风险、简短确认或请求用户确认。

为了避免打扰，第一版每次最多追问 1 到 2 个最关键问题。如果信息已经足够，助手只做简短确认，并给出必要的下一步建议。

自主思考体现在三类行为：

- 追问澄清：把模糊事项变成更明确的目标、截止时间和下一步。
- 判断建议：识别优先级、风险、依赖和推进策略。
- 复盘督办：早晚根据历史记录主动指出停滞、临期、等待他人反馈和高风险事项。

## 算法研发场景支持

用户是算法研发岗，助手需要对技术事项有额外敏感度。

当助手识别到算法研发相关事项时，除通用项目管理问题外，还应提示以下维度：

- 实验/模型任务：数据集版本、baseline、评价指标、实验变量、复现实验记录。
- 线上效果问题：流量范围、渠道差异、数据分布变化、模型版本、特征链路、召回、排序、后处理、埋点变更。
- 需求评估：业务指标、离线指标、上线验收口径、灰度策略。
- Bug/异常排查：触发条件、日志、样本、时间范围、最近变更。
- 跨团队协作：依赖的数据、接口、产品口径、上线窗口、责任人。
- 技术债/优化：收益、风险、回滚方案、对已有实验结论的影响。

助手不直接假设技术结论，而是以“建议确认”“可以优先排查”“这里可能有风险”的方式辅助用户思考。第一阶段不接入代码、日志或实验平台，因此不做自动诊断。

## 数据结构

第一版每条事项保存以下字段：

- 标题
- 原始输入
- 类型：任务、问题、风险、实验、线上异常、会议/沟通、待确认
- 目标
- 截止时间或关注时间
- 状态：待澄清、待推进、等待他人、进行中、已完成、已搁置
- 优先级
- 风险等级
- 相关人/协作人
- 下一步行动
- 技术维度：数据、指标、模型/模块、实验版本、上线/回滚信息
- 最近一次更新
- 对话摘要

第一版不做复杂项目层级，只做事项、标签和状态。使用一段时间后，再评估是否增加项目、里程碑、OKR 或看板。

## 早晚报

上班提醒的目标是挑出今天最该关注的事项，而不是罗列所有待办。候选规则包括：

- 今天或明天到期的事项。
- 高风险事项或线上异常。
- 等待他人反馈过久的事项。
- 昨天新增但仍未明确下一步的事项。
- 算法研发相关的实验、指标、上线风险。

下班总结的目标是帮助用户收束一天。内容包括：

- 今天新增事项。
- 今天有进展的事项。
- 今天完成的事项。
- 仍然缺目标、截止时间或下一步的事项。
- 明天需要优先关注的风险。
- 可用于日报或周报的简短工作记录。

## 隐私与安全

第一阶段默认数据保存在国内云服务器，模型服务优先使用国内大模型服务。

安全要求：

- 钉钉回调接口必须验签。
- 数据库保存任务原文和摘要，但系统不主动读取钉钉聊天或邮箱。
- 日志尽量不记录完整敏感内容，只记录请求 ID、错误类型和必要调试信息。
- 模型密钥、钉钉密钥和数据库密钥必须放在服务器环境变量中，不写进代码。
- 模型调用层应保留 Provider 抽象，避免被单一厂商绑定。

## 错误处理

需要覆盖以下情况：

- 模型无法判断消息意图时，助手应请求用户确认，而不是静默创建错误事项。
- 时间表达模糊时，助手应追问或按默认规则保存为关注时间。
- 模型服务失败时，系统至少保存原始消息，并回复用户“已收到，但智能解析暂时失败”。
- 钉钉消息发送失败时，后端记录可排查日志。
- 数据库写入失败时，助手明确提示记录失败，避免用户误以为已保存。

## 第一版验收标准

使用真实工作流验收，而不是只检查功能列表：

- 用户在钉钉里发送一段模糊工作事项，助手能正确记录并追问关键缺口。
- 用户补充一句进展，助手能关联到已有事项，而不是总是新建。
- 助手能识别算法研发相关事项，并给出数据、指标、实验、上线风险方面的提醒。
- 上班提醒能推送 5 条以内真正值得关注的事项。
- 下班总结能生成当天工作摘要和明日关注点。
- 连续试用 1 周后，用户感到事项不再散落在聊天和脑子里。

## 实施计划建议默认值

实施计划阶段可以基于以下默认值继续细化，除非用户明确调整：

- 云服务器：优先选择阿里云 ECS，便于接入阿里云百炼和钉钉生态。
- 后端技术栈：优先选择 Python + FastAPI，便于快速实现大模型调用、定时任务和结构化数据处理。
- 数据库：第一版优先 PostgreSQL；如果只做单机 MVP，也可以用 SQLite 起步，再迁移到 PostgreSQL。
- 默认模型：优先使用通义千问/Qwen 或阿里云百炼模型服务；DeepSeek、智谱、豆包作为后续评测备选。
- 钉钉接入：优先使用企业内部机器人或应用机器人，支持接收用户消息和主动推送早晚报。
- 早晚报时间：默认按工作日上班前和下班前发送，具体时间在实施计划中确认。
- 第一版命令：建议支持“列出待推进事项”“完成某事项”“今天总结”“更新某事项”四类简单命令。
